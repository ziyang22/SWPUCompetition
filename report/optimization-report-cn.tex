\section{问题背景}

在石油钻井工程中，井下仪器（如测井仪、完井工具）需要通过弯曲、狭窄的井眼轨迹，其通过性直接决定作业成败。传统基于经验的判断方法精度低、计算效率差，难以满足大规模数据处理需求。

本次赛题提供的\textbf{井眼通过能力分析模型}基于三维几何投影算法，可以精确计算仪器最大可通过半径、卡点深度等关键工程参数，但在处理大规模数据时存在显著性能瓶颈，串行计算耗时过长，亟需系统性优化。

\section{问题分析与瓶颈定位}

\subsection{投影法算法原理}

投影法算法沿井眼轨迹以工具长度为窗口滑动，通过三维几何投影计算确定当前窗口的最大可通过直径。算法首先在当前窗口选取上下两个深度平面，然后在井眼轴线周围的锥角范围内采样多个投影方向；对每个投影方向，将上平面的所有三维井壁点投影到下平面，并将投影后的三维点转换为下平面上的二维坐标；随后通过角度分组找出多边形边界点，在网格上搜索最大内切圆，最终得到该窗口的最大可通过直径。若最大直径小于工具直径，则判定仪器在此处卡住。

该算法计算复杂度较高：对于 100 米深度范围，步长 0.5 米，共需要计算 200 个窗口；每个窗口需要采样数十到数百个投影方向；每个投影方向都需要依次进行三维投影变换、三维转二维、边界点选取、最大内切圆搜索；其中最大内切圆搜索本身具有 $O(G^2 N)$ 复杂度，$G$ 为网格分辨率，$N$ 为边界点数量。

\section{优化思路与方案}

我们采用了\textbf{多层递进}的优化策略，从架构级、算法级到指令级逐层优化：

\subsection{第一阶段：Python 到 C/C++ 移植}

原始实现为纯 Python，解释器开销巨大。我们将整个算法完整移植到 C++，保持算法逻辑完全一致：

\begin{itemize}
\item 彻底消除 Python 解释器开销；
\item 使用连续内存布局代替动态 Python 数据结构，提高缓存利用率；
\item 开启编译器最高级别优化 (-O3)；
\item 保持计算结果完全一致，便于对比验证。
\end{itemize}

\textbf{优化效果：} 在最新一轮 Dataset-1 / pass（100m）基准中，197.37s $\rightarrow$ 1.19s，\textbf{加速比 165.9$\times$}。

\subsection{第二阶段：OpenMP 并行化}

每个窗口的各个投影方向之间天然独立，非常适合并行化。我们在外层循环对投影方向添加 OpenMP 并行：

\begin{itemize}
\item 使用线程局部存储保存当前最优半径，避免频繁同步开销；
\item 仅在更新全局最优结果时使用 critical 段；
\item 保持与串行版本完全相同的打破平局规则，确保结果一致。
\end{itemize}

\textbf{优化效果：} 在最新一轮 Dataset-1 / pass（100m）基准中，1.19s $\rightarrow$ 1.22s，\textbf{整体仍处于同一量级}。这一现象说明，当仅启用传统方向级 OpenMP 而未叠加 SIMD 或新的区间并行策略时，性能收益对不同运行批次较为敏感，其价值更多体现在为后续更高层次并行优化提供线程级执行基础。

\subsection{第三阶段：SIMD 向量化与算法微调}

针对热点函数 $\func{max\_inscribed\_circle()}$ 进行底层优化：

\begin{enumerate}
\item \textbf{平方距离优化}：将平方根计算推迟到最后一步，内层循环仅比较平方距离，消除了每次迭代中 900 次昂贵的 sqrt 调用；
\item \textbf{AVX2 SIMD 向量化}：使用 AVX2  intrinsics 同时计算 4 个边界点的距离，充分利用现代 CPU 的 SIMD 执行单元。
\end{enumerate}

\textbf{优化效果：} 在最新一轮 Dataset-1 / pass（100m）基准中，1.22s $\rightarrow$ 0.69s，\textbf{额外加速比 1.77$\times$}，相对于原始 Python 加速 \textbf{286.0$\times$}。这说明平方距离比较与 AVX2 向量化仍然是固定步长路径中最稳定、最有效的底层优化手段。

\subsection{第四阶段：自适应搜索算法（核心创新）}

这是我们提出的核心算法创新，通过动态调整步长大幅减少需要计算的窗口数量：

\subsubsection{核心观察}

在典型的井眼轨迹中，井眼直径随深度增加逐渐缩小。一旦发现某个窗口的最大内切圆直径小于工具直径，即可确定卡点并提前终止计算。因此，我们不需要对每个窗口都进行完整计算，可以通过大步长快速跳过可通行区域。

\subsubsection{自适应搜索算法}

\begin{algorithm}[htbp]
    \caption{自适应步长搜索算法}
    \label{alg:adaptive_search}
    \begin{algorithmic}[1]
        \REQUIRE 起始深度 $S$，结束深度 $E$，最小步长 $\Delta_{\min}$，大步长 $\Delta_{\text{large}}$，工具半径 $R$
        \ENSURE 卡点深度 $D$ 或确认全程通过

        \STATE $current \leftarrow S$
        \WHILE{$current \leq E$}
            \STATE 调用完整算法计算 $current$ 处的最大可通过直径 $d$
            \IF{$d < 2R$}
                \STATE \textbf{回溯精确定位：}从 $current - \Delta_{\text{large}}$ 开始，以 $\Delta_{\min}$ 为步长精确定位卡点
                \STATE 返回精确卡点深度
            \ENDIF
            \IF{最近所有窗口直径均显著大于 $2R$}
                \STATE 增大步长，采用更粗粒度采样
            \ELSE
                \IF{直径接近 $2R$}
                    \STATE 减小步长，采用更细粒度采样
                \ENDIF
            \ENDIF
            \STATE $current \leftarrow current + step$
        \ENDWHILE
        \STATE \textbf{返回}全程通过（未找到卡点）
    \end{algorithmic}
\end{algorithm}

\subsubsection{性能影响}

从工程效果看，自适应搜索的核心收益并不在于“所有场景都更快”，而在于当井段中确实存在失效窗口时，可以用更少的完整计算次数缩小搜索范围，并在回溯阶段精确定位真实卡点。也正因如此，在 cannot\_pass 场景中，自适应搜索通常能体现出更稳定的工程价值；而在 pass 场景下，由于不存在可提前终止的窗口，其收益往往更多体现为搜索策略的鲁棒性，而不是绝对时间上的极限领先。与此同时，最新版本还进一步引入了两类固定步长并行执行方式：其一是\textbf{窗口内部并行}，继续沿用方向级并行思想并显式控制线程数；其二是\textbf{区间任务并行}，先并行计算多个窗口，再按深度顺序提交结果，以保持最早卡点语义与输出顺序一致。最新基准表明，窗口内部并行在当前 CPU 平台上带来了最显著的收益，而区间任务并行则更多验证了并行执行框架与顺序提交机制的正确性。

\subsection{各阶段性能对比}

\subsection{关键优化点总结}

从整体实现路径来看，本项目的性能提升并非来源于单一技巧，而是由多个层次的优化共同叠加而成。首先，在架构层面，将 Python 参考实现迁移至 C/C++ 后，解释器开销与动态对象管理成本被显著压缩，为后续优化奠定了基础。其次，在并行层面，针对投影方向之间天然独立的特点引入 OpenMP，使多核 CPU 的吞吐能力得到有效释放。再次，在热点函数层面，通过平方距离比较与 AVX2 SIMD 向量化降低了内层距离计算的代价。最后，在算法层面，自适应搜索通过减少非关键窗口的完整求解次数，进一步改善了失败场景下的整体效率与定位能力。

换言之，本项目形成的是一条“架构迁移—并行加速—指令级优化—算法级剪枝”逐层递进的优化链条。其意义不仅在于获得更高的加速比，更在于在保持结果一致性的前提下，让程序在不同数据集与不同工况下都具备稳定、可解释且可复现的工程表现。

\section{总结与结论}

综合 2026-03-26 的最新基准结果可以看出，本项目的优化工作已经进入“结果正确 + 并行形态扩展”的新阶段。首先，Python 参考实现与 C++ 版本之间仍然保持明显的数量级差异，这说明将核心算法下沉到底层语言、重组数据布局并减少解释器参与，依然是整个性能提升过程的根本前提。其次，传统 OpenMP 与 SIMD 优化仍然能够稳定地将 100m 级别典型场景压缩到亚秒级，而最新加入的窗口内部并行进一步把 Dataset-1 / pass 的总耗时降低到 0.40s，并在 Dataset-1 / cannot\_pass 场景达到 0.38s，对应约 506.6$\times$ 的总加速比，成为当前版本最优的有效配置。更重要的是，区间任务并行虽然暂未在绝对耗时上超过窗口内部并行，但它已经成功验证了固定步长路径下“区间任务并行 + 顺序提交”的实现方式能够保持最早卡点语义、输出顺序与回归结果一致，这对于后续继续挖掘窗口级任务并行潜力具有直接意义。
